0. 목적
SK㈜ 및 주요 포트폴리오사의 DART 공시를 폴링하여, PR 리스크 관점에서 즉시 대응이 필요한 공시만 선별해 텔레그램으로 알린다. 1차 버전은 공시유형(pblntf_ty) 1차 필터 → 제목(report_nm) 키워드 2차 필터 2단계 구조로 동작한다.

1. 감시 대상 회사 (corp_code 매핑 필요)
DART는 회사명이 아닌 8자리 corp_code로 조회한다. 아래 회사명에 대해 corpCode.xml(DART 고유번호 파일)에서 corp_code를 매핑해 config/companies.py에 상수로 둔다.


# config/companies.py (예시 — corp_code는 매핑 후 채울 것)


WATCH_COMPANIES = {


    "SK주식회사":   "",   # SK㈜ (지주)


    "SK이노베이션": "",


    "SK텔레콤":     "",


    "SK스퀘어":     "",


    "SK하이닉스":   "",


    # 필요 시 추가: 매각/추가투자 대상 포트폴리오사


}


* 1차 가동은 SK㈜ 단일 회사로 시작해 검증 후 확장 권장.

3. 제목 키워드 2차 필터 (report_nm)
1차로 받아온 공시 목록에서 제목(report_nm)에 아래 키워드가 포함된 건만 통과시킨다. 부분 문자열 매칭(in), 공백·특수문자 변형 대비해 정규화(공백 제거) 후 비교 권장.


# config/keywords.py


TRIGGER_KEYWORDS = {


    # --- D: 지분공시 (행동주의·지분이동 조기경보) ---


    "D": [


        "대량보유",        # 5%룰 (주식등의 대량보유상황보고서)


        "주식등의",


        "임원ㆍ주요주주",  # 특정증권등 소유상황보고서


        "임원·주요주주",


        "소유상황",


        "의결권",


    ],


    # --- B: 주요사항보고 (리밸런싱·중대 의사결정) ---


    "B": [


        "양수도",          # 영업/주식 양수도


        "양수",


        "양도",


        "출자증권",


        "타법인",


        "자기주식",        # 자사주 취득·처분


        "유상증자",


        "무상증자",


        "감자",


        "합병",


        "분할",


        "주식교환",


        "소송",            # 소송 등의 제기·신청


        "채무보증",


        "회사채",


        "전환사채",


        "신주인수권",


        "영업정지",


        "해산",


        "파산",


    ],


    # --- I: 거래소공시 (해명 요구 = PR 즉각 반응) ---


    "I": [


        "조회공시",        # 거래소 조회공시 요구/답변


        "풍문",


        "보도",            # 풍문 또는 보도에 대한 해명

        "분기보고서",


    ],


}


매칭 방식: 회사 corp_cls가 코스닥/유가에 무관하게 제목 기준으로 거른다. 오탐 발생 시 키워드를 제거·추가하며 튜닝(설정 파일 분리한 이유).

통과된 공시는 등급을 매겨 텔레그램 메시지 머리말(이모지)로 구분한다.


등급
	조건

	발견 즉시 발송
	🟡 보통
	pblntf_ty == A
	1일 1회 일괄
	⚪ 참고
	그 외 통과 건
	일괄

5. 중복 방지 (상태 관리)
* DART 각 공시의 고유 식별자 rcept_no(접수번호) 를 키로 사용.
* 이미 알린 rcept_no는 재발송하지 않는다.
* 1차 버전 저장소: 로컬은 state/seen.json, Railway 배포 시 Railway Volume 경로(예: /data/seen.json)에 저장.
* 구조 예시:


{


  "seen": ["20260620000123", "20260620000456"],


  "last_run": "2026-06-20T09:00:00+09:00"


}


* 매 실행 시: 신규 rcept_no만 추출 → 알림 → seen에 추가 → 저장.
* seen 리스트는 최근 N건(예: 2000건)만 유지해 무한 증가 방지.

6. 텔레그램 알림 메시지 4단 구조 (PR 템플릿)
각 트리거 공시마다 아래 4단으로 구성한다. (3·4단은 v1.1에서 Claude API로 자동 생성, v1은 1·2단 + 링크만)


🔴 [공시 포착] {회사명}


1) 무엇이 공시됐나


   · 제목: {report_nm}


   · 유형: {유형 한글} | 접수: {rcept_dt}


   · 원문: https://dart.fss.or.kr/dsaf001/main.do?rcpNo={rcept_no}


2) PR 영향도   ← v1은 키워드 기반 한 줄, v1.1은 Claude 생성


   · {예: 행동주의 지분 확대 가능성 / 매각 본계약 / 거래소 해명 요구}


3) 예상 질문   ← v1.1


4) 권고 대응 메시지   ← v1.1


v1은 1)·2)단과 원문 링크까지 구현해 "정확히 잡아서 즉시 알린다"를 먼저 완성한다. 3)·4) 자동 생성은 트리거·알림이 안정화된 뒤 Claude API를 붙인다.

7. DART list API 호출 메모
* 엔드포인트: https://opendart.fss.or.kr/api/list.json
* 핵심 파라미터:
   * crtfc_key = os.getenv("DART_API_KEY")
   * corp_code = 감시 회사
   * bgn_de = 어제(KST), end_de = 오늘(KST) — 최근 2일 조회로 24시간 공백 없이 커버. 중복은 rcept_no로 제거되므로 범위 겹침 무해.
   * pblntf_ty = B / D / I (유형별로 분리 호출하거나 루프)
   * page_count = 100, sort_mth = desc
* 응답 status == "000" 이 정상. "013"은 데이터 없음(정상 처리).
* 하루 호출 한도 주의: 회사 × 유형 × cron 빈도가 곱해지므로, cron 간격(예: 평일 30분)과 유형 분리 호출 수를 곱해 한도 내인지 확인.

8. 1차 가동 범위 (Definition of Done)
v1은 아래까지만 완성한다.


* SK㈜ corp_code 매핑
* B·D·I 유형 list API 호출 성공
* 키워드 2차 필터 통과 건 추출
* rcept_no 기준 중복 제거 (seen.json)
* 텔레그램으로 1)·2)단 + 원문 링크 발송
* Railway cron 30 8 * * 1-5 (UTC=KST 17:30, 평일) 배포 + Volume에 seen.json 영속화


3)·4)단 자동 생성, 다중 회사 확장, A·F·J 유형 추가는 v2로 분리.
